당신은 견적 시스템의 신뢰성·수치 안정성을 검토하는 시니어 엔지니어 겸 여행상품 애널리스트입니다. 아래 설계를 비판적으로 검토하고 구체적으로 고쳐 주세요. 칭찬 말고 결함·리스크를 짚어 주세요.

## 배경
한국발 MICE 단체연수 견적 자동계산기입니다. 견적 단가에는 여러 '계수(배수)'가 곱해집니다(기준 1.0):
- seasonFactor: 출발 '월'별 시즌 계수(예: 성수기 1.20, 비수기 0.88). 항공·유류·호텔에 곱함.
- leadFactor: 예약 리드타임 계수(임박 1.25 ~ 조기 0.92). 항공·유류에만.
- peakFactor: 날짜 단위 피크(골든위크 1.35 등). 항공·유류에.
- hotelPeakFactor: 호텔이 받는 피크. 현재 1 + (peakFactor−1)×0.8 (호텔은 항공 피크의 80%만).

이 계수들은 지금 코드 상수(하드코딩)라, 조정하려면 개발자가 배포해야 합니다. 이를 '관리자 페이지에서 담당자가 조정 가능'하게 만들려 합니다(P2b). 단, 아직 실제 계약가 실측 데이터가 없어 자동 캘리브레이션은 불가하고, 지금은 '담당자 도메인 판단으로 배포 없이 조정'이 목적입니다.

## 설계(스칼라 노브부터 — 안전 우선)
표(달력·시즌표)를 통째로 편집하는 UI는 복잡·위험하니, 1차로는 '전역 보정 강도' 스칼라 4개만 관리자에서 조정 가능하게 합니다. 각 노브는 기존 계수의 '진폭'만 완만/과격하게 조절:

- 공통 수식: adjusted = 1 + (baseFactor − 1) × strength
  - strength = 1.0 → 기존과 완전 동일(무변화). 0이면 계수 무력화(항상 1.0). 1보다 크면 진폭 확대.
- 노브 목록/기본값:
  1. seasonStrength = 1.0  (시즌 계수 진폭; 항공·유류·호텔에 반영)
  2. leadTimeStrength = 1.0 (항공 리드타임 진폭)
  3. peakStrength = 1.0     (날짜 피크 진폭; 항공·유류)
  4. hotelPeakWeight = 0.8  (호텔이 받는 피크 비중; 이건 진폭이 아니라 '비중' 자체. hotelPeakFactor = 1 + (rawPeak−1)×hotelPeakWeight)

- 저장/전달: 관리자가 저장하면 서버 DB(app_settings 같은 KV)에 넣고, 공개 계산기가 페이지 로드시 /api/rates로 받아 코드 기본값 위에 덮음. 값이 없거나 fetch 실패하면 기본값(=현재 동작). 폴백-우선.
- 방어: 서버·클라이언트 양쪽에서 숫자·범위 검증. 이상값이 견적을 폭주시키지 않도록 클램프.

## 검토 요청(구체적으로)
1. 이 4개 노브가 '담당자가 실무에서 조정하고 싶어할 최소 유용 세트'로 적절합니까? 빠진 것(예: fx 클램프 범위, 비수기 하한)이나 불필요한 게 있습니까?
2. 각 노브의 '안전 범위(min~max clamp)'를 얼마로 두는 게 좋습니까? strength가 음수/0/과대일 때의 위험은? (예: strength 3이면 성수기 1.20이 1.60까지 튐)
3. seasonStrength가 항공·유류·호텔 '전부'에 동시에 적용되는 게 맞습니까, 아니면 항목별로 분리해야 합니까?
4. peakStrength(항공 피크 진폭)와 hotelPeakWeight(호텔 피크 비중)를 '독립'으로 두는 설계(peakStrength는 호텔에 영향 없음)가 직관적입니까, 혼란스럽습니까?
5. 수식 adjusted = 1 + (f−1)×strength 방식의 수치적 함정(예: 여러 계수 곱해질 때 상호작용, 음수 진입)이 있습니까?
6. 당신이 추천하는 '최종 노브 목록 + 각 기본값 + clamp 범위' 표를 딱 떨어지게 제시해 주세요.

한국어로, 6번은 코드에 옮길 수 있게 명확히.
